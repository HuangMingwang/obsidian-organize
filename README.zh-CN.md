# 🗂 obsidian-organize

> 面向 Obsidian 笔记的引导式自动化整理：自动分类、补结构、更新索引，但不失去控制

![关系图谱](assets/graph.png)

▲ 整理后的 Obsidian 知识图谱 — PARA 分类 + 双向链接自动生成

## 为什么需要 obsidian-organize？

笔记越记越多，但——

- 📁 文件全堆在一个文件夹里，找不到
- 🏷 想打标签但懒得手动分类
- 🔗 知道笔记之间有关联，但从来不建链接
- 📑 MOC 索引想维护但总是忘记更新

**obsidian-organize 自动处理重复劳动，但不替你乱猜。**
对 Claude 说「整理笔记」，它会：

1. 读懂内容，自动归入 PARA 文件夹
2. 生成完整的 frontmatter（标题、分类、标签、摘要）
3. 维护托管结构内容，比如相关笔记区块和 MOC
4. 遇到拿不准的笔记时汇报出来，而不是静默乱猜

首次执行 `整理` 或 `生成` 时，它会先给出一个简短的 vault 配置向导，确认后再写入 `.obsidian-organize.yml`。

如果当前 vault 使用 git，单篇请求使用 1 次 git commit；批量请求最多使用 2 次 commit：先提交逐篇本地变更，再提交共享图谱更新。如果你拒绝 `git init`，skill 也可以继续执行，但没有基于 git 的回滚。

## ✨ 三种模式

### 重写 — 「帮我精简这篇笔记」

去掉冗余和废话，保留核心逻辑和代码示例。原文自动备份到 Archives。重写只改正文，不碰元数据、分类和链接。

加上「只清理格式」只做格式清理（多余空格、标点混用、标题层级），不改内容。

### 整理 — 「整理笔记」

自动分类 → 生成元数据 → 移动到对应文件夹 → 维护托管区块（`frontmatter`、`## 相关笔记`、MOC）。除非你明确先执行重写，否则整理不会改正文主体内容。

### 生成 — 「生成一篇关于 XXX 的笔记」

支持四种输入源：

- 💬 给个主题，从零生成
- 🗣 把当前对话整理成笔记
- 🔗 从 URL 抓取内容生成
- 📋 粘贴一段内容整理成笔记

生成会先产出草稿，再把这份草稿送进标准的整理流程。

## 🚀 快速开始

### 安装

```bash
npx skills add https://github.com/HuangMingwang/obsidian-organize
```

### 使用

在 Claude Code 中 `cd` 到你的 Obsidian vault 目录，然后说：

| 你说 | 效果 |
|------|------|
| 「整理笔记 xxx.md」 | 分类、补全元数据、移动文件、更新相关笔记和 MOC |
| 「精简笔记 xxx.md」 | 只重写内容，原文备份 |
| 「重写笔记，只清理格式」 | 只清理格式，不改内容 |
| 「生成一篇关于 Docker 的笔记」 | 先生成草稿，再进入标准整理流程 |
| 「重写笔记 Inbox/」 | 按小批次批量重写目录下的笔记，并持续汇报进度 |
| 「整理这个文件夹」 | 批量整理目录下的笔记；拿不准的先跳过，最后统一更新链接和 MOC |

首次执行 `整理` 或 `生成` 时，会先给出一个有边界的配置向导，确认后写入 `.obsidian-organize.yml`。

## 📦 大批量处理

当你让 obsidian-organize 处理整个文件夹或非常多笔记时，它会先统计候选笔记数量，告诉你处理范围，然后按小批次执行，并持续汇报进度。

拿不准的笔记会先跳过并在最后统一汇报，不会卡住整批任务。`相关笔记`、反向链接和 MOC 这类共享图谱内容会在最后统一回写。如果运行环境支持并行，也只会并行逐篇本地处理，不会并行写共享文件。批量请求最多使用 2 次 commit，而不是逐篇单独 commit。

## 📖 背后的方法论

本项目基于三套经典笔记管理方法：

- **PARA** — 按 Projects / Areas / Resources / Archives 四层分类，让每条笔记都有归属
- **MOC（Map of Contents）** — 为每个领域维护索引页，替代传统文件夹层级导航
- **Zettelkasten（卡片笔记法）** — 通过双向链接建立笔记间的语义关联

obsidian-organize 自动处理这三套方法论里最重复的维护动作，你仍然保留结构控制权。

## 🔒 安全机制

- 启用 git 时，单篇请求使用 1 次 git commit；批量请求最多使用 2 次 commit
- 重写前自动备份原文到 Archives
- 单篇分类拿不准会询问你；批量里拿不准的会跳过并汇报
- `![[image.png]]` 这类 wiki 图片会保留原位，不会被强迁移
- 不会删除任何文件
